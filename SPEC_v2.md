# SPEC v2 — Lampa-Trakt Plugin

Спецификация поведения плагина v2. Пишется до кода. Язык — человеческий, не технический. Архитектура переработана относительно v1 (см. `../v1_SPEC.md` для исходной версии и журнал v1 в `../v1_README.md`).

## Философия

**Trakt — единственный источник правды.** Плагин — клиент Trakt, не store. Любое отображение строится из текущего состояния Trakt; любое действие пользователя транслируется в вызов Trakt API.

**Плагин не вмешивается в нативные структуры Lampa.** В отличие от v1, мы НЕ пишем в `Lampa.Favorite`, не маппим нативные папки на Trakt-сущности, не слушаем `Favorite.listener`. У нас собственный пункт в левом меню навигации, собственный экран с собственным представлением, собственное in-memory хранилище отображения.

**Это устраняет родовой класс v1-багов.** В Lampa.Favorite одновременно пишут юзер, наш плагин и `cub.red` (нативная синхронизация Lampa-аккаунта); порядок недетерминирован, гонки маскируются эвристиками. v2 этой гонки не видит — мы рядом, не в ней.

Последствие: в v2 нет `enforceStatusExclusivity`, `verifyStatusFoldersClean`, `findRealCard`, `_recentWrites` — эти конструкции были компенсацией для гонки в Lampa.Favorite, которой больше нет. Защиты `_pendingLampaMark/_pendingLampaUnmark` тоже не делаем на Фазах 1–2, так как синхронизация эпизодов отложена (см. §«Решения, отложенные на потом»); если вернёмся — защиты вернутся вместе с подсистемой.

**То, что остаётся** — компенсация другого рода: stale-окно самого Trakt API (5–15 минут между write и появлением записи на read-эндпоинтах). Это серверная задержка, она не зависит от клиентской архитектуры. Защитный буфер Pending Ops в v2 остаётся, в упрощённой форме (см. §«Pending Ops»).

## Модель папок

В v2 у плагина **пять взаимоисключающих статусов**. Один item (фильм или сериал) имеет ровно один статус из пяти, либо не имеет статуса вообще (тогда не отображается ни в одной нашей папке).

| Статус | Описание | Что в Trakt |
|---|---|---|
| **Закладка** | Хочу посмотреть, ещё не начал | Watchlist + не сработало ничего другого |
| **Смотрю** | Сериал начат, есть что смотреть прямо сейчас | `progress.completed > 0 && progress.next_episode != null` |
| **Просмотрено** | Фильм просмотрен, или сериал догнан и закрыт | Фильм в history, ИЛИ сериал догнан + статус `ended/canceled` |
| **Пауза** | Сериал догнан, ждём новых серий | Сериал догнан + статус `returning series / in production / planned` |
| **Брошено** | Не интересно, скрыть из основных списков | Union трёх ячеек Trakt (см. §«Логика Брошено») |

Никаких тегов, никаких двойных статусов. Если Trakt-данные приводят к нескольким статусам одновременно (например, item и в Watchlist, и в history) — побеждает один по приоритету классификатора, второй молча игнорируется.

«Пауза» в Trakt API нативно отсутствует — это наша производная классификация поверх стандартных Trakt-полей. Пользователь её не выбирает вручную (см. §«Действия пользователя»).

## Классификатор статуса

Для каждого item-а из Trakt-данных применяется первая срабатывающая проверка. Порядок проверок = приоритет.

```
1. item в hidden-union (см. §«Логика Брошено»)
   → Брошено

2. item — фильм:
   2.1. в Trakt history (есть запись в /sync/watched/movies или plays > 0)
        → Просмотрено
   2.2. в Trakt watchlist
        → Закладка
   2.3. иначе → нигде

3. item — сериал:
   3.1. progress.completed == 0
        3.1.1. в Trakt watchlist → Закладка
        3.1.2. иначе → нигде
   3.2. progress.completed > 0:
        3.2.1. progress.next_episode != null
               → Смотрю
        3.2.2. progress.next_episode == null + show.status ∈ {ended, canceled}
               → Просмотрено
        3.2.3. progress.next_episode == null + show.status ∈ {returning series, in production, planned}
               → Пауза
```

**Ключевой сигнал — `progress.next_episode == null` («догнан»).** Это ровно тот сигнал, по которому Trakt UI кладёт сериал в Continue Watching. Не сравниваем `aired` vs `completed`, не используем `last_episode` (последняя *просмотренная*, а не вышедшая). Это бесплатная замена VIP-эндпоинта `up_next_nitro`.

**Эксклюзивность статусов гарантируется автоматически** — каждое решение классификатора возвращает ровно один статус из пяти. В отличие от v1, не нужно отдельно «защищать» от двойных статусов: их источник (mark в `Lampa.Favorite`) у нас не используется.

## Действия пользователя

Принцип: **действие в плагине = вызов Trakt API + локальная отметка в нашем буфере Pending Ops**. Lampa-плагин ничего долгосрочно у себя не хранит; на следующем reconcile отображение пересчитывается из Trakt + накладывается pending op в активном TTL-окне.

Действия, доступные через UI плагина (см. §«UI-модель» Фаза 2):

| Действие | Вызов Trakt API |
|---|---|
| Поставить **Закладку** | `POST /sync/watchlist` |
| Снять **Закладку** | `POST /sync/watchlist/remove` |
| Отметить «**Смотрю**» (сериал) | `POST /sync/history` на S01E01 (минимальная подсказка Trakt'у: completed > 0, next_episode указывает дальше; классификатор отдаёт «Смотрю») |
| Отметить «**Просмотрено**» (фильм) | `POST /sync/history` с movies |
| Отметить «**Просмотрено**» (сериал) | `POST /sync/history` со всеми вышедшими эпизодами (`progress.seasons[].episodes[]`) |
| Снять «**Просмотрено**» (фильм) | `POST /sync/history/remove` с movies |
| Снять «**Просмотрено**» (сериал) | `POST /sync/history/remove` со всеми вышедшими эпизодами |
| Снять «**Смотрю**» (сериал) | `POST /sync/history/remove` на S01E01 (зеркало add) — отменяет нашу подсказку. Реальную пользовательскую историю за пределами S01E01 НЕ трогаем |
| Поставить «**Брошено**» | Тройной параллельный write (см. §«Логика Брошено») |
| Снять «**Брошено**» | Тройной параллельный remove |

**«Пауза» как пользовательское действие отсутствует.** Это автоматическая производная классификатора. В сайдбаре выбора папки на детальной карточке (Фаза 2) пункта «Пауза» нет. Если юзер хочет «отложить сериал» — кладёт в **Закладки** (хочу посмотреть позже) или в **Брошено** (не интересно).

**Резолв типа карточки (movie / show)** — через Trakt `/search/tmdb/:id`, не по полям карточки Lampa. В нашем UI карточки часто приходят «тощими» (без `method`, `card_type`, `number_of_seasons`); в `/sync/watchlist` шлём только корректный массив (movies или shows, не оба) — это предотвращает баг 0.5.x–0.8.3 v1 («один tmdb_id занят и фильмом, и сериалом, Trakt добавил обе сущности»).

**Watchlist exclusivity при переходе в статусную папку.** Когда пользователь явно ставит item в Смотрю / Просмотрено / Брошено — мы дополнительно шлём `POST /sync/watchlist/remove`, чтобы Watchlist не оставался поверх. Trakt сам делает это для Смотрю/Просмотрено через настройку «Auto remove from watchlist on watch» (срабатывает при scrobble), но мы дублируем явным remove'ом для надёжности. Для Брошено настройка Trakt не помогает (scrobble не происходит) — поэтому remove критичен. Обратное действие (поставил Закладку на уже просмотренное) — НЕ снимает статус: пользователь сам решает.

## UI-модель

UI делится на три фазы. Фаза 1 — обязательный MVP, без неё плагин не запускается. Фаза 2 — write-actions, доступные после Фазы 1. Фаза 3 — отложенный ресёрч.

### Фаза 1: страница со списком

**Точка входа.** Свой пункт «Trakt» (или другой нейминг по выбору) в левом меню Lampa. Технически — DOM-инъекция `<li class="menu__item selector">` в `$('.menu .menu__list').eq(0)` (паттерн из `trakt_by_lampame.js:5877–5980`). Триггер — событие `app:ready` (паттерн `Lampa.Listener.follow('app', cb)`).

**Открытие.** При `hover:enter` на пункт меню — сразу `Lampa.Activity.push({ component: 'trakt_v2_main', title: 'Trakt' })`. Без промежуточного `Lampa.Select.show` (этот промежуточный сайдбар у `trakt_by_lampame` неудобен; v2 от него отказывается).

**Раскладка экрана.** Как нативное Избранное Lampa (см. `../страница_избранного.png`):

- Сверху — горизонтальный ряд кнопок-якорей со счётчиками: `Закладки 5 / Смотрю 12 / Просмотрено 47 / Пауза 3 / Брошено 14`. Клик по кнопке скроллит к соответствующей секции.
- Ниже — пять секций одна под другой. Каждая — заголовок + горизонтальный ряд карточек. Внутри секции — мини-разбивка по типу контента (Фильмы N / Сериалы M), как в нативном Избранном.

**Карточки.** Стандартный `Lampa.Card`, обогащение через TMDB (постеры, локализованные названия, рейтинги). Кеш TMDB-данных в памяти на время сессии.

**Скролл и фокус.** Через стандартный `Lampa.Controller`. Вертикальный скролл проходит через все пять секций; внутри секции — горизонтальный скролл по карточкам.

**Пустые секции.** Если в папке нет items — секция отображается с заголовком и пометкой «пусто», но не пропускается. Пользователю важно видеть пять одинаковых рядов.

**Нет токена / нет данных.** Если `Lampa.Storage.get('trakt_token')` пустой — экран показывает заглушку «Авторизуйтесь в Trakt через настройки плагина TraktTV». Никакой собственной OAuth-кнопки на Фазе 1 не делаем.

### Фаза 2: write-actions

**Кнопка «Trakt» на детальной карточке.** Через `Lampa.Listener.follow('full', cb)` ловим открытие full-card view, добавляем кнопку в `.full-start__button-wrapper` (паттерн из `trakt_by_lampame.js:5637`). Клик открывает кастомный сайдбар (см. ниже).

**Долгий тап в обзоре нашей страницы.** Аналогично — на карточке в нашей секции долгий тап (или контекстное меню в Tizen) открывает тот же сайдбар.

**Кастомный сайдбар выбора папки.** `Lampa.Select.show({ title, items, onSelect })` с пунктами:
- **Закладки** (если уже в Закладках — пункт читается как «убрать из Закладок»)
- **Смотрю** (если уже в Смотрю — «убрать из Смотрю»)
- **Просмотрено** (если уже в Просмотрено — «убрать из Просмотрено»)
- **Брошено** (если уже в Брошено — «убрать из Брошено»)

Пункта «Пауза» в сайдбаре нет (см. §«Действия пользователя»).

При клике по пункту — оптимистично обновляем локальное представление (карточка едет в новую папку немедленно), параллельно шлём write в Trakt API, кладём pending op в Storage. На следующем reconcile состояние пересчитывается из Trakt; pending op в активном TTL-окне форсит ожидаемое.

### Фаза 3: подмена нативного сайдбара (отложено, ресёрч)

В Lampa на детальной карточке существует свой нативный сайдбар выбора папок (от `Lampa.Favorite`), отличный от нашего кастомного. Это даёт пользователю «прыжок» между двумя UX (наш кастомный сайдбар vs нативный сайдбар Lampa.Favorite). Цель Фазы 3 — устранить этот прыжок.

Варианты для ресёрча (выбор после Фаз 1+2):

- **(a) Подмена.** Перехватить открытие нативного сайдбара и подставить наш контент.
- **(b) Отключение.** Отключить нативный сайдбар на нашей карточке, оставить только кастомный (через скрытие нативной кнопки или подмену её обработчика).
- **(c) Расширение.** Добавить наши папки в нативный сайдбар как дополнительные пункты, оставив нативные на месте. Менее радикально, но возможны логические противоречия.

Откладывается до стабильной работы Фаз 1+2.

## Границы с `trakt_by_lampame.js`

Плагин `trakt_by_lampame` остаётся зависимостью v2 на старте (как и в v1). Используется **точечно**.

**Берём:**
- **OAuth flow** — кнопка логина в его настройках, device code flow, polling, refresh-токен.
- **Хранение токена** — `Lampa.Storage.get('trakt_token')`, `Lampa.Storage.get('trakt_refresh_token')` и связанные ключи (`trakt_token_created_at`, `trakt_token_expires_in`, `trakt_token_expires_at`).
- **Scrobble эпизодов** — на плеере (`Lampa.Player.listener`).
- **UI авторизации** — никаких своих экранов логина не делаем.

**НЕ берём:**
- Никакой кодовой связи. Только чтение `Lampa.Storage`-ключей.
- НЕ используем `window.TraktTV.api.*` — все Trakt-запросы делаем сами.
- НЕ используем VIP-эндпоинты (`up_next_nitro` и т.п.).
- Его меню-пункты, его компоненты (`trakt_watchlist`, `trakt_upnext`, `trakt_my_lists` и т.д.) — это его UI, не наш.
- Его кнопку «Manage lists» на детальной карточке — у нас своя.

**Слушатель `Lampa.Storage.listener.follow('change', cb)`** — следим за изменением `trakt_token`, чтобы переинициализировать наш экран при логауте/смене аккаунта.

**Финальная независимость** (свой `client_id` Trakt-приложения + свой прокси) — отложена до первых рабочих результатов v2. Без неё плагин не «полностью независимый», но это решается одним днём инфраструктуры (Cloudflare Worker / тонкий node-сервис с `client_id` в env) и не блокирует разработку.

## Сетевой слой и Trakt API

**Базовый URL прокси:** `https://apx.lme.isroot.in/trakt` (тот же, что использует `trakt_by_lampame`). Прокси на сервере подставляет `trakt-api-key` (client_id, под которым выпущен токен) и форвардит на `api.trakt.tv`.

**Заголовки для всех запросов:**
```
Content-Type: application/json
trakt-api-version: 2
Authorization: Bearer <trakt_token>   // только для авторизованных эндпоинтов
```

**Транспорт:** `fetch` (с fallback на `XMLHttpRequest` если потребуется для совместимости с Tizen — проверим на устройстве).

**Обработка ошибок:**
- `401` → токен протух/невалиден. На Фазе 1 — показать заглушку «авторизуйтесь». Refresh-логика остаётся за `trakt_by_lampame` (он сам обновляет токен в Storage; мы реагируем на изменение через `Storage.listener`).
- `429` (rate limit) → exponential backoff, retry до 3 раз. Trakt rate limit 1000/5мин — на типичную нагрузку плагина (десятки запросов на reconcile) запас на порядок.
- `5xx` → retry до 2 раз с задержкой; затем — показать пользователю «Trakt временно недоступен», не прерывать рендеринг (показываем кешированное состояние).
- `2xx` без тела или с пустым ответом — нормально для write-эндпоинтов.

**Рейт-контроль:** не нужен на типичной нагрузке. Если в логах увидим близость к 1000/5мин — добавим in-memory очередь.

**Ключевые эндпоинты для read-пути:**
- `GET /sync/watchlist/movies?extended=full` — фильмы в Watchlist.
- `GET /sync/watchlist/shows?extended=full` — сериалы в Watchlist.
- `GET /sync/watched/movies?extended=full` — фильмы в history (для Просмотрено).
- `GET /sync/watched/shows?extended=full` — сериалы с прогрессом (для Смотрю / Просмотрено / Пауза).
- `GET /shows/:trakt_id/progress/watched` — детальный прогресс сериала (для `next_episode`). Вызывается на каждый сериал из watched-списка с `completed > 0`.
- `GET /users/hidden/progress_watched?type=show` — ячейка hpw для Брошено.
- `GET /users/hidden/dropped?type=show` — ячейка hdr для Брошено.
- `GET /users/me/lists/:list_id/items` — кастомный список Брошено.
- `GET /users/me/lists` — список user-lists для селектора в настройках Брошено.
- `GET /search/tmdb/:tmdb_id?type=movie,show` — резолв TMDB → Trakt id и определение типа.

**Ключевые эндпоинты для write-пути:**
- `POST /sync/watchlist` / `POST /sync/watchlist/remove` — Закладки.
- `POST /sync/history` / `POST /sync/history/remove` — Смотрю / Просмотрено.
- `POST /users/hidden/progress_watched` / `/remove` — hpw.
- `POST /users/hidden/dropped` / `/remove` — hdr.
- `POST /users/me/lists/:list_id/items` / `/remove` — кастомный список.

## Pending Ops

**Зачем нужны.** Trakt API после успешного 2xx write отдаёт устаревшие данные на read-эндпоинтах в течение 5–15 минут (эмпирически измерено в v1). Без буфера: пользователь поставил item в папку → reconcile сразу после действия → Trakt ещё не показывает изменение → плагин «откатывает» отображение → через 10 минут возвращается. UX ломается.

**Решение** — TTL-буфер недавних writes в `Lampa.Storage`. Каждая запись:

```js
{
  id: <tmdb_id>,
  type: 'movie' | 'show',
  folder: 'book' | 'look' | 'viewed' | 'paused' | 'thrown',
  action: 'add' | 'remove',
  ts: <unix_seconds>
}
```

Внутренние коды папок (`book` / `look` / `viewed` / `paused` / `thrown`) — для совместимости с v1 кодовой базой, в UI пользователь видит русские имена.

**После каждого успешного 2xx write** — добавляем запись.

**На каждом reconcile** (открытие нашего экрана):
1. Тянем состояние Trakt параллельными запросами.
2. Применяем классификатор → desired state по пяти папкам.
3. Накладываем pending ops с TTL-окном:
   - `action: 'add'` и Trakt ещё не показывает item в этой папке → форсим наличие.
   - `action: 'remove'` и Trakt ещё показывает item в этой папке → форсим отсутствие.
4. Pending ops старше TTL — удаляем из Storage.

**Параметр TTL:** 15 минут по умолчанию, настраивается в настройках (15 мин / 5 мин / 1 мин / 10 сек / Выключено) — как в v1 0.8.3+. «Выключено» обходит буфер: ничего не сохраняется, на diff ничего не накладывается. Полезно для отладки.

**Свойства механизма:**
- Trakt остаётся единственным долгосрочным источником правды; pending ops — временное наложение.
- Идемпотентен: если Trakt догнал до истечения окна — наложение бесшумно совпадает с реальностью.
- В отличие от v1: нет `_recentWrites` (нет двойных эмитов `Favorite.listener` — у нас вообще нет этого listener'а). Защиты для эпизодов (`_pendingLampaMark/_pendingLampaUnmark`) тоже не нужны пока синхронизация `Lampa.Timeline` отложена (см. §«Решения, отложенные на потом»).

**Эксклюзивность через pending ops.** Когда пользователь ставит item в папку X, мы дополнительно записываем pending op `remove` в остальные четыре папки с тем же id. Это гарантирует, что в TTL-окне item не «появится» ни в какой другой папке из устаревших Trakt-данных. Семантика: `add` в X → принудительно ТОЛЬКО в X.

## Логика Брошено

Логика берётся из v1 0.10.0 практически как есть. Подробное описание — в `../v1_SPEC.md` §«Логика Брошено». Краткое содержание для v2:

**Trakt не имеет одной нативной ячейки «брошено».** Разные клиенты пишут в разные структуры:
- Moviebase «Stop watching» → `/users/hidden/progress_watched` (`hpw`).
- Trakt-веб «Drop show» → `/users/hidden/dropped` (`hdr`).
- Custom list — обычный пользовательский список (как UX-метка с visual badge на постере).

Hidden-ячейки и custom list **не зеркалят друг друга автоматически**.

**Стратегия v2 (как в v1 0.10.0):**

- **Тройной write при «Брошено»:** параллельный POST в `hpw` + `hdr` + custom list.
- **Зеркальный тройной remove при снятии Брошено.**
- **Read-union при reconcile:** параллельный GET всех трёх ячеек, union по tmdb id. Item считается thrown, если найден хотя бы в одной.
- **Канонизация при reconcile:** для каждого item-а в union, который есть НЕ во всех трёх ячейках, fire-and-forget POST'ом дописываем недостающие. Так v2 выступает «мостом» между MB и Trakt-веб, которые сами по себе не зеркалят.
- **Throttle канонизации:** `_recentCanonization[cell:tmdb_id]` с TTL 15 мин — не дописываем один и тот же id в одну и ту же ячейку чаще раза в TTL.

**Custom list — настройка пользователя.** Плагин не создаёт его автоматически. В настройках v2 — селектор «Папка Trakt для статуса Брошено» с подгрузкой через `GET /users/me/lists`. Хранится `ids.trakt` (числовой стабильный идентификатор). Без выбора — деградированный режим: пишем только в `hpw` + `hdr`, в MB / Trakt-веб не будет визуального значка.

**Приоритет в классификаторе.** Hidden-ячейки НЕ сбрасывают watched-историю в Trakt — брошенный сериал по-прежнему в `/sync/watched/shows`. Поэтому проверка thrown идёт **до** классификации look/viewed/paused (см. §«Классификатор статуса», шаг 1).

**Фильмы.** Hidden-эндпоинты не поддерживают `type=movie` — фильмы попадают только в кастомный список. Без настроенного списка фильм в Брошено локально не удержится.

**Pending Op для thrown** — особо важна, так как hidden-эндпоинты Trakt самые медленные.

## Триггеры синхронизации

- **При первом открытии нашего экрана** в текущей сессии — полный reconcile (читаем все Trakt-эндпоинты, строим desired state).
- **При повторном открытии** в той же сессии — повторный reconcile (полный или частичный — параметр оптимизации, на старте делаем полный).
- **При смене токена** (`Storage.listener` отлавливает изменение `trakt_token`) — пересинхронизация при следующем открытии экрана.
- **Никаких фоновых таймеров.** Reconcile только по триггеру открытия.
- **Ручной кнопки «синхронизировать»** не нужно (на Фазе 1; можно добавить в настройки если потребуется для отладки).

## Что НЕ реализуется в v2

- **Лайки, рекомендации, рейтинги Trakt** — не отображаем.
- **Отдельная папка «История просмотров»** — фильмы попадают в Просмотрено, отдельной папки не нужно.
- **TraktTV-домашние ряды на главном экране Lampa** — пусть остаются у `trakt_by_lampame`.
- **Чужие custom-листы пользователя** (кроме «Брошено» list) — не отображаем.
- **Поля `reset_at`** из progress — игнорируем как в v1.
- **Оценки/ratings** — не синхронизируются.
- **Запланировано (как отдельный статус)** — не вводим. «Хочу посмотреть» = Закладка, «отложил на паузу» = Закладка или Брошено по выбору пользователя.

## Решения, отложенные на потом

Подсистемы, по которым решение принципиально *не закрыто* — не реализуются на Фазах 1–2, но и не выкинуты из скоупа окончательно. Возвращаемся к ним после первого MVP на Tizen, когда станет понятно, что базовая часть стабильно работает.

### Синхронизация отметок эпизодов с `Lampa.Timeline`

В v1 0.9.0–0.9.3 это была трёхнаправленная синхронизация:
- *Lampa → Trakt (отметка):* при воспроизведении — через scrobble `trakt_by_lampame`. При ручной отметке — listener на `Timeline.update`, percent ≥ 90 → `POST /sync/history`.
- *Lampa → Trakt (снятие):* listener percent=0 → `POST /sync/history/remove`. Закрывало пробел «снял в Lampa → осталось просмотренным в Trakt».
- *Trakt → Lampa:* pull-reconcile при открытии экрана — `/sync/watched/shows` + `/shows/:id/progress/watched`, сверка с `Timeline.file_view` (ключ — `Utils.hash([season, sep, episode, original_name||original_title])`), дописывание недостающего.

Защитные слои `_pendingLampaMark` и `_pendingLampaUnmark` (TTL ~60 сек) защищали ручные отметки и снятия от мгновенного отката reconcile-ом, пока Trakt API ещё не отразил write на read-эндпоинтах.

**На Фазах 1–2 не делаем.** Серии помечаются стандартным `Timeline` + scrobble через `trakt_by_lampame` (только при воспроизведении). Последствия:
- Отметка серии в другой Lampa или в Trakt-вебе **не появится** в нашей Lampa автоматически — каждое устройство ведёт свой `Timeline`.
- Ручная отметка серии в Lampa без воспроизведения **не уйдёт** в Trakt.
- Снятие отметки в Lampa **не уйдёт** в Trakt.

**Точка возврата.** После Фазы 2, когда базовый плагин подтверждён на Tizen и UX папок отлажен. Возвращение зависит от поведения cub.red относительно `Timeline.file_view`:
- Если cub.red **не** трогает `Timeline` — переносим логику v1 0.9.3 практически как есть (~150–200 строк).
- Если cub.red трогает `Timeline` — деградированный вариант: только Trakt → Lampa pull (показываем в Lampa то, что отмечено в Trakt). Ручная двунаправленность не делается из-за тройной гонки. ~60–80 строк.

**Что уходит из v1:**

- Запись в `Lampa.Favorite` целиком (все Favorite-листенеры, Favorite.add/remove из плагина, mark-операции).
- `enforceStatusExclusivity` / `verifyStatusFoldersClean` / `findRealCard` — компенсации для гонки в Lampa.Favorite.
- `_recentWrites` — защита от двойных эмитов `Favorite.listener` (этого listener'а у нас нет).
- `_pendingLampaMark` / `_pendingLampaUnmark` для эпизодов — синхронизация `Lampa.Timeline` отложена (см. §«Решения, отложенные на потом»); защиты вернутся вместе с подсистемой, если решим её делать.
- `clearBookMark`, `markOwn`-защита — компенсации `Favorite.listener` петли.
- Reconcile типа карточки в Watchlist через `cardIsShow` — у нас он строится напрямую из Trakt-ответа.

**Что остаётся концептуально (но переписывается с нуля):**

- Pending Ops от stale-окна Trakt — упрощённая форма (см. §«Pending Ops»).
- Брошено-триплет с канонизацией — почти как есть из v1 0.10.0.
- TMDB-обогащение карточек.

- Резолв типа карточки (movie / show) через `/search/tmdb/:id` для тощих карточек.

**Что появляется нового:**

- Свой Activity component (`Lampa.Component.add('trakt_v2_main', factory)`).
- Свой layout главного экрана (5 секций как в нативном Избранном).

## Глоссарий

- **watchlist** (Trakt) — список «хочу посмотреть», без различия фильм/сериал.
- **progress** (Trakt) — объект с `aired`, `completed`, `next_episode`, `seasons[]` и т.п. для конкретного сериала. Возвращается `/shows/:trakt_id/progress/watched`.
- **scrobble** (Trakt) — событие start/pause/stop во время воспроизведения, которое автоматически помечает эпизод просмотренным. В v2 — целиком за `trakt_by_lampame`.
- **canonization** (наш термин) — процесс «дописать недостающие ячейки» в трёх Trakt-target'ах для Брошено: если карточка есть в hpw + list, но нет в hdr — fire-and-forget POST в hdr.
- **stale-окно** (наш термин) — серверная задержка Trakt API между принятым write и его отражением на read-эндпоинтах того же объекта. Эмпирически 5–15 мин.
- **Pending Ops** (наш термин) — TTL-буфер недавних write-операций, который во время stale-окна Trakt подменяет недопроявившиеся изменения и не даёт reconcile «откатить» только что сделанное действие.
- **reconcile** — пересчёт desired state папок из текущего состояния Trakt + наложение pending ops.
- **desired state** — таблица «какой item в какой папке должен быть отображён», построенная классификатором из Trakt-данных и накладок pending ops.
- **hpw / hdr** (наши коды) — `/users/hidden/progress_watched` / `/users/hidden/dropped`.

## Архитектура на будущее

В `../v1_SPEC.md` описаны два варианта серверной эволюции (A: тонкий воркер для канонизации и кеша; B: VPS как primary store с outbox + pull-loop). Эти варианты остаются в силе для v2 — переход на сервер не зависит от того, реализуем ли мы клиентскую часть через Lampa.Favorite или через свой UI.

**Когда стоит начинать сервер.** Не сейчас. Сначала добиваем чисто клиентскую v2 на одном устройстве, проверяем на Tizen, валидируем UX. Точка перехода — когда подключится второй-третий клиент и упрёмся в межустройственный рассинхрон. К тому моменту клиентская логика должна быть стабильной.

**Что точно станет проще на сервере:**
- Pending Ops растворяются (stale-окно Trakt не видно клиенту в варианте B).
- Канонизация Брошено уходит в cron, не привязана к открытию экрана.
- Эпизод-синхронизация между устройствами (если решим её делать) становится возможной без ручных трюков.

Подробнее — в `../v1_SPEC.md` §«Архитектура v2: серверный воркер».
